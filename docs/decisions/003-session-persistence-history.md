# 003 — 会话持久化与历史记录

**日期**: 2026-07-19

**状态**: 已通过

## 背景

心率数据仅在内存中维护环形缓冲区（900 点），应用退出后数据丢失。需要将每次应用启动到退出的完整心率数据持久化，并提供历史浏览功能。同时，需要在图表中明确标出数据断连（无信号）的时间段。

## 决定

1. **会话模型**：从成功连接 BLE 设备开始，到设备断开连接（含完全退出程序）结束为一个会话。会话 ID 为连接时刻的 ISO 风格字符串（文件系统安全格式 `YYYY-MM-DDTHH-mm-ss`），存储在 `AppData/sessions/{id}.json`。
2. **持久化策略**：`session` store 每 5 秒从 `heartRateHistory` store 拉取新数据点，增量写入会话文件。应用退出时最终刷盘并标记结束时间。
3. **断连标记**：当两次心率采样间隔超过 2 秒时，`heartRateHistory` store 在断连时刻插入 `value: 0` 的数据点。
4. **0 值渲染**：图表按值是否为 0 将数据点拆分为连续段，非零段用主色 + 渐变填充，零值段用灰色虚线（`#d4d4d4`，无填充）。Y 轴最小值调整为 0。
5. **历史浏览**：`/history` 列表页展示所有历史会话（日期、时长、数据点数）；`/session/:id` 详情页复用 `HeartRateChart` 组件展示该会话曲线。

## 原因

- **增量写入**：避免每秒写盘造成 IO 压力，5 秒间隔兼顾安全性和性能。
- **0 值标记而非缺省**：比在渲染层推算断连更直观可靠，且 0 值天然参与统计计算。
- **会话独立文件**：比单文件追加更易管理（删除、加载单个会话），文件体积可控。

## 影响

- 正面：完整心率历史可追溯；断连区间可视化清晰
- 负面：长时间会话可能产生大文件（1 小时 ≈ 3600 个数据点 ≈ 200KB）
- 后续：需要考虑会话文件的删除功能、数据导出、以及旧会话自动清理策略
